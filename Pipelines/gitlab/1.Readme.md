# CI/CD Pipeline — Java Microservice

## Overview

This GitLab CI/CD pipeline builds, scans, and deploys a Java WAR-based microservice running on WildFly (JBoss) into three Kubernetes environments on AWS EKS.

| Environment | K8s Namespace    | Cluster              | Trigger                  |
|-------------|------------------|----------------------|--------------------------|
| **BUILD**   | `digital-build`  | `aws-np-cluster`     | Push to `release/*`      |
| **ST**      | `digital-st`     | `aws-np-cluster`     | Tag `X.Y.Z-RCn-ST`      |
| **PT**      | `digital-pt`     | `aws-np-pt-cluster`  | Tag `X.Y.Z-RCn-PT`      |

---

## Pipeline Stages

```
┌──────────────────────── Quality Gate ────────────────────────┐
│                                                              │
│  install ──┬── dependency-check                              │
│            ├── code-quality (SonarQube)                      │
│            ├── sast-scan (SpotBugs)                          │
│            └── publish-war (Nexus)                           │
│                                                              │
└──────────────────────────────────────────────────────────────┘
                          │
            ┌─────────────┼─────────────┐
            ▼             ▼             ▼
     docker-build     (ST tag)      (PT tag)
            │                           │
     deploy-build    deploy-st    docker-build-pt
                                        │
                                    deploy-pt
```

### Stage Details

| # | Stage              | What it does                                                                 |
|---|--------------------|------------------------------------------------------------------------------|
| 1 | **install**        | `mvn clean install -DskipTests` — compiles the project and produces the WAR. |
| 2 | **dependency-check** | OWASP Dependency-Check scans third-party libraries for known CVEs.         |
| 3 | **code-quality**   | SonarQube analysis for bugs, code smells, and coverage.                      |
| 4 | **sast-scan**      | SpotBugs static analysis for security and correctness issues.                |
| 5 | **publish-war**    | Deploys the WAR artifact to the Nexus repository.                            |
| 6 | **docker-build**   | Builds a Docker image with Kaniko (no Docker daemon) and pushes to ECR.      |
| 7 | **deploy-build**   | Rolls out the new image to the BUILD K8s namespace.                          |
| 8 | **deploy-st**      | Rolls out the existing image to the ST K8s namespace (tag-triggered).        |
| 9 | **docker-build-pt**| Builds a separate image using the PT base image and pushes to ECR.           |
| 10| **deploy-pt**      | Rolls out the PT image to the PT K8s namespace.                              |

## Trigger Rules

### 1. Merge Request → `release/*`

When a developer opens an MR from `feature/*` into a `release/*` branch, only the **quality gate** stages run (install through publish-war). This gives fast feedback before merging.

### 2. Push to `release/*`

After the MR is merged, the push event runs the full pipeline: quality gate **+** docker build **+** deploy to BUILD. This is the main integration path.

### 3. Tag `X.Y.Z-RCn-ST`

Creating a tag like `1.2.0-RC1-ST` promotes the already-built image to the **ST** (System Test) namespace. No rebuild occurs — it reuses the image from the release push.

### 4. Tag `X.Y.Z-RCn-PT`

Creating a tag like `1.2.0-RC1-PT` triggers a fresh Docker build using a **separate PT base image** (`BasePTImage` variable) and deploys to the **PT** (Performance Test) namespace on a dedicated cluster.

## Required CI/CD Variables

Set these in **Settings → CI/CD → Variables**:

| Variable                  | Description                                      | Example                                      |
|---------------------------|--------------------------------------------------|----------------------------------------------|
| `baseImage`               | Base Docker image for BUILD/ST                   | `wildfly:26.1-jdk17`                         |
| `BasePTImage`             | Base Docker image for PT environment             | `wildfly:26.1-jdk17-pt`                      |
| `CI_REGISTRY_IMAGE`       | ECR repository URI (auto-set or manual)          | `123456789.dkr.ecr.eu-west-2.amazonaws.com/myapp` |
| `CI_REGISTRY_IMAGE_LATEST`| ECR repo for latest/alias tags                   | Same as above or separate                    |
| `CI_PT_REGISTRY_IMAGE`    | ECR repository URI for PT images                 | `123456789.dkr.ecr.eu-west-2.amazonaws.com/myapp-pt` |
| `CI_PT_PROJECT_NAME`      | Project name override for PT tagging             | `myapp-pt`                                   |
| `SONAR_HOST_URL`          | SonarQube server URL                             | `https://sonar.example.com`                  |
| `SONAR_TOKEN`             | SonarQube authentication token                   | *(masked)*                                   |
| `NEXUS_REPO_URL`          | Nexus repository URL for WAR publishing          | `https://nexus.example.com/repository/maven-releases` |
| `CLONE_REPO_TOKEN`        | Token for cloning shared Dockerfile repo         | *(masked)*                                   |

## Docker Image Strategy

Images are built with [Kaniko](https://github.com/GoogleContainerTools/kaniko) (no privileged Docker-in-Docker required) and tagged as follows:

| Tag pattern                                        | Purpose                           |
|----------------------------------------------------|-----------------------------------|
| `SNAPSHOT-<project>-<commit-sha>`                  | Unique, immutable per commit      |
| `<project>-ST`                                     | Floating "latest ST" alias        |
| `<project>-BE`                                     | Floating "latest backend" alias   |

## How to Use

### Deploy to BUILD (automatic)

```bash
# Merge your feature branch into the release branch — pipeline runs automatically.
git checkout release/1.2
git merge feature/my-feature
git push origin release/1.2
```

### Promote to ST

```bash
git tag 1.2.0-RC1-ST
git push origin 1.2.0-RC1-ST
```

### Promote to PT

```bash
git tag 1.2.0-RC1-PT
git push origin 1.2.0-RC1-PT
```

## Key Improvements over Previous Pipeline

1. **YAML anchors** (`&kaniko_base`, `&k8s_deploy_base`) eliminate duplicated configuration.
2. **Centralised variables** for image tags — single source of truth, no string duplication.
3. **Parallel quality gates** — dependency-check, code-quality, and sast-scan run concurrently after install.
4. **Rollout status checks** — `kubectl rollout status --timeout=300s` makes the pipeline fail if a deployment doesn't stabilise.
5. **Maven cache** — `.m2/repository` is cached across pipelines to speed up builds.
6. **Artifact expiry** — WARs expire after 1 hour, reports after 7 days, keeping storage lean.
7. **Retry on infrastructure failures** — auto-retries once for runner crashes or timeouts.
8. **Removed redundant Dockerfile clone** — the Dockerfile is expected in the repo (or fetched once via a shared job).
9. **Clear stage naming** — stages map directly to what they do, not internal jargon.
