# CI/CD Pipeline — Node.js UI Microservice

## Overview

This GitLab CI/CD pipeline builds, lints, scans, and deploys a Node.js front-end application served via Apache httpd into three Kubernetes environments on AWS EKS.

| Environment | K8s Namespace    | Cluster              | Trigger                  |
|-------------|------------------|----------------------|--------------------------|
| **BUILD**   | `digital-build`  | `aws-np-cluster`     | Push to `release/*`      |
| **ST**      | `digital-st`     | `aws-np-cluster`     | Tag `X.Y.Z-RCn-ST`      |
| **PT**      | `digital-pt`     | `aws-np-pt-cluster`  | Tag `X.Y.Z-RCn-PT`      |

---

## Pipeline Stages

```
┌──────────────────────── Quality Gate ────────────────────────┐
│                                                              │
│  install ── lint ── build ──┬── code-quality (SonarQube)     │
│                             └── fortify-scan (manual)        │
│                                                              │
└──────────────────────────────────────────────────────────────┘
                          │
            ┌─────────────┼─────────────┐
            ▼             ▼             ▼
     docker-build     (ST tag)      (PT tag)
            │                           │
     deploy-build    deploy-st    docker-build-pt
                                        │
                                    deploy-pt
```

### Stage Details

| # | Stage              | What it does                                                                  |
|---|--------------------|-------------------------------------------------------------------------------|
| 1 | **install**        | `npm install` — fetches all Node.js dependencies from the private Nexus registry. |
| 2 | **lint**           | `npm run lint` — runs ESLint/TSLint to catch code style and syntax issues.    |
| 3 | **build**          | `npm run build` — produces the production `dist/` bundle.                     |
| 4 | **code-quality**   | SonarQube scan for bugs, code smells, and coverage against the quality gate.  |
| 5 | **fortify-scan**   | Fortify SCA static analysis — runs on-demand (manual trigger), uploads to SSC.|
| 6 | **docker-build**   | Kaniko builds a Docker image (httpd + dist/) and pushes to ECR.              |
| 7 | **deploy-build**   | Rolls out the new image to the BUILD K8s namespace.                          |
| 8 | **deploy-st**      | Rolls out the existing image to the ST K8s namespace (tag-triggered).        |
| 9 | **docker-build-pt**| Builds a Docker image for the PT environment and pushes to ECR.              |
| 10| **deploy-pt**      | Rolls out the PT image to the PT K8s namespace.                              |

## Trigger Rules

### 1. Merge Request → `release/*`

When a developer opens an MR from `feature/*` into a `release/*` branch, the **quality gate** stages run (install → lint → build → code-quality). Fortify is available as a manual action. No Docker build or deploy happens.

### 2. Push to `release/*`

After the MR merges, the push event runs the full pipeline: quality gate **+** Docker build **+** deploy to BUILD.

### 3. Tag `X.Y.Z-RCn-ST`

Creating a tag like `1.2.0-RC1-ST` promotes the already-built image to the **ST** (System Test) namespace. No rebuild — it reuses the image from the release push.

### 4. Tag `X.Y.Z-RCn-PT`

Creating a tag like `1.2.0-RC1-PT` triggers a full install → lint → build → Docker build and deploys to the **PT** (Performance Test) namespace on a dedicated cluster.

## Required CI/CD Variables

Set these in **Settings → CI/CD → Variables**:

| Variable                  | Description                                      | Example / Note                               |
|---------------------------|--------------------------------------------------|----------------------------------------------|
| `baseImage`               | Apache httpd base Docker image                   | `httpd-patched-2.4.66` (pre-configured)      |
| `CI_REGISTRY_IMAGE`       | ECR repository URI                               | `123456789.dkr.ecr.eu-west-2.amazonaws.com/app` |
| `CI_REGISTRY_IMAGE_LATEST`| ECR repo for floating latest tag                 | Same or separate repo                        |
| `SONAR_LOGIN`             | SonarQube authentication token                   | *(masked)*                                   |
| `SSC_URL`                 | Fortify Software Security Center URL             | `https://fortify.example.com/ssc`            |
| `CI_TOKEN_SC`             | Fortify SSC auth token                           | *(masked)*                                   |
| `CLONE_REPO_TOKEN`        | Token for cloning shared Dockerfile repo         | *(masked)*                                   |

## Docker Image Strategy

Images are built with [Kaniko](https://github.com/GoogleContainerTools/kaniko) (no Docker daemon) using an Apache httpd base image. The `dist/` build output is copied into the httpd document root.

| Tag pattern                                 | Purpose                           |
|---------------------------------------------|-----------------------------------|
| `SNAPSHOT-<project>-<commit-sha>`           | Unique, immutable per commit      |
| `<project>-latest`                          | Floating "latest" alias           |

## How to Use

### Deploy to BUILD (automatic)

```bash
# Merge your feature branch into the release branch — pipeline runs automatically.
git checkout release/1.2
git merge feature/my-feature
git push origin release/1.2
```

### Promote to ST

```bash
git tag 1.2.0-RC1-ST
git push origin 1.2.0-RC1-ST
```

### Promote to PT

```bash
git tag 1.2.0-RC1-PT
git push origin 1.2.0-RC1-PT
```

### Trigger Fortify Scan

Navigate to the pipeline in GitLab UI and click the **Play** button on the `Fortify_Scan` job.

## Key Improvements over Previous Pipeline

1. **YAML anchors** (`&kaniko_base`, `&k8s_deploy_base`, `&rules_mr_and_release`) eliminate duplicated configuration blocks.
2. **Centralised variables** for image tags — `IMAGE_REF` is defined once, not assembled in every script.
3. **Explicit `needs` graph** — stages declare their dependencies so GitLab can parallelise code-quality alongside the build path.
4. **Rollout status checks** — `kubectl rollout status --timeout=300s` fails the pipeline if a deployment doesn't stabilise within 5 minutes.
5. **Artifact expiry** — `dist/` expires after 1 hour, Fortify FPR after 7 days, keeping storage lean.
6. **Retry on infrastructure failures** — auto-retries once for runner crashes or timeouts.
7. **Clean stage naming** — stages map directly to their function (`docker-build`, `deploy-build`) instead of internal shorthand.
8. **Fortify artifacts preserved** — `results.fpr` is saved as a downloadable artifact for audit.
9. **NPM registry as a variable** — easy to change without editing multiple script blocks.
